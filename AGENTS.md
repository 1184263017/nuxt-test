# Codex 仓库协作指南


## Codex 工作原则

Codex 在本仓库中应以工程协作为目标，不做无依据猜测，不做顺手重构，不扩大改动范围。

- 先读代码再判断：实现前优先检索相关页面、接口、路由、状态、权限和工具函数。
- 简单优先：使用现有模式解决当前问题，不为单一场景提前设计抽象层。
- 外科手术式修改：只改与需求直接相关的文件和代码行，避免格式化无关文件。
- 目标驱动验证：每次改动都要明确可验证的完成条件，并运行最相关的检查。
- 尊重脏工作区：不要回滚、覆盖或整理他人改动；发现无关变更时忽略，发现冲突时先说明。
- 不顺手重构：除非需求要求或修复必须，否则不要重命名、迁移目录、统一风格或清理历史代码。

## 验收标准

一次 Codex 交付至少应满足：

- 需求中的核心行为已实现，或明确说明无法实现的阻塞原因。
- 变更范围与需求一致，没有无关文件改动。
- TypeScript、ESLint、构建或人工验证结果清楚可追溯。
- 高风险链路已被检查，未发现明显回归。
- 用户可见文案、注释和说明符合中文约定。
- 最终回复包含变更摘要、验证结果和必要的风险提示。

## 不确定项处理

遇到不理解的业务逻辑、框架 API、依赖用法或架构背景时，不要凭猜测实现。

优先顺序：

1. 检索本仓库代码、`README.md`、配置文件和相邻模块。
2. 查找同类页面或同类接口的既有实现。
3. 仍不明确时，使用 MCP 工具查证：`context7` 用于等第三方库官方文档；`deepwiki` 用于理解外部仓库、复杂模块关系或架构背景。
4. 如仍存在业务歧义，向用户提出简短、具体的问题，并说明当前可安全推进和不可安全推进的部分。

## Git 与工作区

- 修改前后注意工作区可能已有用户改动。
- 不要执行 `git reset --hard`、`git checkout --` 等会丢弃改动的命令，除非用户明确要求。
- 不要把无关文件加入提交。
- 不要因为 lint 或 format 造成大面积无关格式化。
- 除非用户明确要求，否则 Codex 不主动提交代码。
- 如必须提交，提交信息使用简短 conventional 前缀和中文摘要，例如 `fix: 修复线索列表权限判断`。

## Pull Request 要求

PR 描述应包含：

- 变更摘要。
- 已执行的验证命令。
- 关联 issue 或任务编号，如有。
- UI 变更的截图或录屏，如适用。
- 配置、路由、权限或接口契约变化的影响说明。

## 安全与配置

- 不要提交密钥、Token、账号、私有证书、环境专属凭据或真实客户数据。
- 不要将本地调试地址、临时代理、个人路径写入生产配置。
- 修改 API 地址、鉴权逻辑、请求签名、第三方 ID 或环境变量时，应说明影响范围。
- 调整依赖时避免引入无关 lockfile 变更。
- 不要为了本地调试把生产配置改成临时值后遗留在代码中。

## 编码前思考
不要假设。不要隐藏困惑。呈现权衡。
明确说明假设 — 如果不确定，询问而不是猜测
呈现多种解释 — 当存在歧义时，不要默默选择
适时提出异议 — 如果存在更简单的方法，说出来
困惑时停下来 — 指出不清楚的地方并要求澄清

## 简洁优先
用最少的代码解决问题。不要过度推测。
不要添加要求之外的功能
不要为一次性代码创建抽象
不要添加未要求的"灵活性"或"可配置性"
不要为不可能发生的场景做错误处理
如果 200 行代码可以写成 50 行，重写它
检验标准： 资深工程师会觉得这过于复杂吗？如果是，简化。

## 精准修改
只碰必须碰的。只清理自己造成的混乱。
编辑现有代码时：
不要"改进"相邻的代码、注释或格式
不要重构没坏的东西
匹配现有风格，即使你更倾向于不同的写法
如果注意到无关的死代码，提一下 —— 不要删除它
当你的改动产生孤儿代码时：

删除因你的改动而变得无用的导入/变量/函数
不要删除预先存在的死代码，除非被要求
检验标准： 每一行修改都应该能直接追溯到用户的请求。

## 目标驱动执行
定义成功标准。循环验证直到达成。
将指令式任务转化为可验证的目标：
不要这样做...	转化为...
"添加验证"	"为无效输入编写测试，然后让它们通过"
"修复 bug"	"编写重现 bug 的测试，然后让它通过"
"重构 X"	"确保重构前后测试都能通过"
对于多步骤任务，说明一个简短的计划：
1. [步骤] → 验证: [检查]
2. [步骤] → 验证: [检查]
3. [步骤] → 验证: [检查]
